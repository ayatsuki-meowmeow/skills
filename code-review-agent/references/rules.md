# コードレビューエージェント — ルール本体

## なぜこの構造か

単一のレビューエージェントに「良さそうに見てくれ」と丸投げすると、次のいずれかで詰まる:

- **観点の偏り** — プロンプトを書いた人の関心が強い観点だけ拾い、ほかを見落とす
- **観点の被り** — バグ / 品質 / 保守性が同じ指摘を別々に上げてノイズになる
- **主観の混入** — 「なんとなく気になる」レベルの nitpick が並ぶ
- **仕様との照合が抜ける** — code のロジック中心に見て、design.md との乖離を見逃す

これを避けるため、レビューを **独立レンズの並列 + confidence scoring** に構造化する。各レンズは責務を分離し、被らない切り口で見る。指摘は Haiku で採点し、閾値未満を機械的に切る。

参照実装は Anthropic 公式 code-review プラグイン（`~/.claude/plugins/cache/claude-plugins-official/code-review/`）。多レンズ + 採点 + 閾値の骨格をそこから借り、design.md / impl.md 連携と working diff レビュー向けに書き直したものが本 skill。

## 前提と位置づけ

- 本 skill は **implement-review-loop の step 3（レビュー）の実行系**。単独起動もできるが、想定用途は implement-review-loop の内側
- 対象タスクの `design.md` / `impl.md` が揃っている（なければ先に design-impl-docs skill）
- サブエージェント（Task / Agent ツール）が利用できる
- レビュー対象は **working diff**（GitHub PR ではない）。PR のレビューは公式 code-review プラグインが担う

## 全体フロー

```
[入力] design.md path / impl.md path / 変更ファイル一覧 / 変更概要
   ↓
1. 5 レンズを並列委譲（各レンズは独立 subagent）
   - Lens 1: 外部仕様遵守（design.md）
   - Lens 2: 内部整合性（impl.md）
   - Lens 3: バグ・エッジケース
   - Lens 4: デグレ影響
   - Lens 5: 保守性・可読性
   ↓
2. 各レンズは「指摘 + 分類ヒント + 該当箇所」のリストを返す
   ↓
3. 全指摘を集約し、各指摘に対して Haiku で confidence scoring を並列実行
   ↓
4. score < 80 を切る（閾値は必要に応じて調整可）
   ↓
5. 生き残った指摘を implement-review-loop に返す
   - 未解決の疑問（仕様への疑問など）はレポート先頭に別立てで出す
```

## 5 レンズの詳細

各レンズは独立サブエージェントとして起動する。共通の委譲プロンプト要素は subagent-orchestration の「委譲の仕方」に従う（役割 / コンテキスト / タスク / エスカレーション指示 / 境界）。

### Lens 1: 外部仕様遵守（design.md）

**目的:** design.md に書かれた仕様と、実装された振る舞いのズレを検知する。要件の実装過不足、API 契約の逸脱、UI 挙動・エラー時の動作の相違など、**外部から観測できる振る舞い**に絞る。

**拾うもの:**

- design.md に書かれた要件のうち、実装されていない or 明らかに過剰な部分
- API の入出力形状 / エラー時レスポンスが design.md と異なる
- UI の状態遷移や表示条件が design.md と異なる

**拾わないもの:**

- 内部実装の話（Lens 2 に任せる）
- design.md に書かれていない領域の判断（未解決の疑問として上げる）

**分類ヒント初期値:** 仕様・設計

**モデル:** Opus（仕様と実物の照合は判断が重い）

**委譲プロンプト雛形:**

```
あなたは外部仕様遵守レビューエージェントです。
コンテキスト: <docs/<feature>/design.md> をまず読んでください。impl.md は参照しません。

タスク:
以下の変更差分が design.md の仕様に対して過不足なく実装されているかレビューしてください。
観点は「外部から観測できる振る舞い」に限定します（API 契約 / UI 挙動 / エラー時の動作 / 要件の網羅）。
内部実装の議論はしないでください（別レンズが見ます）。

変更差分:
<変更ファイル一覧・概要 or 差分>

出力形式: rules.md の「出力形式」に従うこと。分類ヒントの初期値は「仕様・設計」。
design.md に記述がない領域の判断が必要になった場合は、勝手に判断せず「未解決の疑問」として出力してください。
成果物末尾に自身のコンテキスト使用率（概算）を含めてください。
```

### Lens 2: 内部整合性（impl.md）

**目的:** impl.md の記載と実装の実物のズレを検知する。impl.md に書かれた構造・技術判断・実装状況と現在のコードが食い違っていないか。

**拾うもの:**

- impl.md に書かれた構造・モジュール分割と実物のズレ
- impl.md の「技術的判断」の理由と実装の齟齬
- 実装状況（完了 / 進行中 / 未着手）の記載漏れ・不整合

**拾わないもの:**

- 外部仕様との照合（Lens 1）
- コードの品質問題（Lens 5）

**分類ヒント初期値:** 実装判断

**モデル:** Sonnet

**委譲プロンプト雛形:**

```
あなたは内部整合性レビューエージェントです。
コンテキスト: <docs/<feature>/impl.md> をまず読んでください。design.md は参照しません。

タスク:
以下の変更差分が impl.md の「構成」「技術的判断」「実装状況」と整合しているかレビューしてください。
impl.md に書かれた内容と実物のコードのズレ・記載漏れが主な観点です。
仕様（design.md）との照合はしないでください（別レンズが見ます）。

変更差分:
<変更ファイル一覧・概要 or 差分>

出力形式: rules.md の「出力形式」に従うこと。分類ヒントの初期値は「実装判断」。
成果物末尾に自身のコンテキスト使用率（概算）を含めてください。
```

### Lens 3: バグ・エッジケース

**目的:** 差分内で完結する誤り・エッジケースの見落としを検知する。

**拾うもの:**

- null / undefined / 空配列 / ゼロ / 境界値・オフバイワンの見落とし
- 例外・エラー時の分岐漏れ（未 catch / 未 handling）
- 並行性・順序依存の問題（差分内で判断できる範囲）
- 早期 return / ガード節の抜け

**拾わないもの:**

- 呼び出し元への影響（Lens 4）
- スタイル・命名の指摘（Lens 5）
- linter / typechecker が catch する系（false positive リスト参照）

**分類ヒント初期値:** 実装判断

**モデル:** Sonnet

**委譲プロンプト雛形:**

```
あなたはバグ検知レビューエージェントです。
コンテキスト: 変更差分に限定して読んでください。呼び出し元や過去コミットは追わないでください（別レンズが見ます）。

タスク:
以下の変更差分に含まれるバグ・エッジケースの見落としを指摘してください。
観点: null / undefined / 境界値 / 例外分岐 / 並行性・順序依存 / ガード節。
linter や typechecker が catch する類の指摘（型ミス・import 忘れ・formatter）は挙げないでください。
スタイル・命名の好みは挙げないでください（別レンズが見ます）。

変更差分:
<変更ファイル一覧・概要 or 差分>

出力形式: rules.md の「出力形式」に従うこと。分類ヒントの初期値は「実装判断」。
「なんとなく気になる」レベルは含めない。明確な失敗シナリオを示せる指摘のみ挙げてください。
成果物末尾に自身のコンテキスト使用率（概算）を含めてください。
```

### Lens 4: デグレ影響

**目的:** 差分の「外」への影響を検知する。既存インターフェースの変更が呼び出し元を壊していないか、過去 PR / commit で議論された内容と矛盾していないか。

**拾うもの:**

- 変更されたシグネチャ・エクスポート・ファイル配置の呼び出し元への影響
- 過去 PR / commit のコメントで意図された制約と反する変更
- git blame / history 上で「これは変えないでね」的な履歴があるコードへの変更

**拾わないもの:**

- 差分内で完結するバグ（Lens 3）
- スタイル・可読性（Lens 5）

**分類ヒント初期値:** 実装判断

**モデル:** Sonnet

**委譲プロンプト雛形:**

```
あなたはデグレ検知レビューエージェントです。
コンテキスト: 変更差分に加えて、変更されたインターフェースの呼び出し元、必要なら該当ファイルの git blame / 過去コミットも参照してください。

タスク:
以下の変更差分が既存機能・呼び出し元を壊していないかレビューしてください。
観点: シグネチャ変更の影響範囲 / 削除・改名されたエクスポートの他所からの参照 / 過去 PR / commit で議論された制約への違反。
差分内で完結するバグは挙げないでください（別レンズが見ます）。

変更差分:
<変更ファイル一覧・概要 or 差分>

出力形式: rules.md の「出力形式」に従うこと。分類ヒントの初期値は「実装判断」。
過去コミット参照時は該当 SHA / メッセージを引用してください。
成果物末尾に自身のコンテキスト使用率（概算）を含めてください。
```

### Lens 5: 保守性・可読性

**目的:** 差分内のコードの読みやすさ・保守しやすさに関する指摘。

**拾うもの:**

- 意味を捉えづらい命名（読み手に事前知識を要求しすぎ）
- 差分内での重複（3 回以上の類似コード）
- 複雑な条件分岐（ネスト過剰 / 早期 return で潰せる形）
- altitude 混在（低レベル操作と高レベルの意図が同じ関数に同居）

**拾わないもの:**

- 単発の好みレベル（コーディング規約 skill / linter に載っていないもの）
- 既存コードのスタイル（差分外は指摘しない）
- 命名の細かい bikeshedding（明確に意味を損ねている場合のみ挙げる）

**分類ヒント初期値:** 実装判断

**モデル:** Sonnet

**委譲プロンプト雛形:**

```
あなたは保守性・可読性レビューエージェントです。
コンテキスト: 変更差分に限定して読んでください。

タスク:
以下の変更差分の可読性・保守性に関する指摘をしてください。
観点: 命名（意味が捉えづらいもの）/ 差分内 3 回以上の類似コード / 過剰なネスト / altitude 混在。
コーディング規約 skill / linter で catch できる類の指摘は挙げないでください（別で拾われます）。
単発の「好み」レベルの命名 bikeshedding は挙げないでください。

変更差分:
<変更ファイル一覧・概要 or 差分>

出力形式: rules.md の「出力形式」に従うこと。分類ヒントの初期値は「実装判断」。
成果物末尾に自身のコンテキスト使用率（概算）を含めてください。
```

## 共通 False Positive リスト

以下は **どのレンズでも指摘しない**。プロンプトに含めるか、集約段階で機械的にフィルタする:

- **linter / formatter / typechecker が catch する系**: 型エラー / import 忘れ / インデント / セミコロン / 未使用変数 / stylistic な形式指摘
- **pre-existing issue**: 変更差分の外にある既存の問題（本タスクのスコープ外）
- **好みの nitpick**: senior engineer が実務では指摘しない類の単発スタイル指摘
- **意図的に silenced されているもの**: `// eslint-disable-next-line` / `// TODO(reason)` などが付いているもの
- **明らかに意図された挙動変更**: PR / 差分の趣旨として妥当な範囲の変更
- **coding-rule skill でカバーされている領域**: プロジェクトに該当 skill があるならそちらに任せる（例: `neverthrow-coding-rules` があるなら Result 型の扱いはそちらで catch されるので本レビューで挙げない）

## 出力形式

各レンズは以下の JSON 相当の構造で返す（Markdown リストでも可、下記はスキーマ）:

```
{
  "lens": "外部仕様遵守" | "内部整合性" | "バグ・エッジケース" | "デグレ影響" | "保守性・可読性",
  "findings": [
    {
      "file": "path/to/file.ts",
      "line": 42,
      "summary": "何が問題か（1 文）",
      "detail": "詳細と、実際に失敗するシナリオ",
      "suggestion": "どう直すべきかの案（なくても可）",
      "category_hint": "仕様・設計" | "実装判断"
    }
  ],
  "unresolved_questions": [
    "仕様に対する疑問（勝手に判断せず、ユーザーへエスカレーションすべきもの）"
  ],
  "context_usage": "自己申告 %（概算）"
}
```

指摘がなければ `findings: []` を明示的に返す。`concerns: 「概ね良好」` のような曖昧な返しは禁止。

## Confidence Scoring（Haiku）

全レンズの結果を集約したあと、**指摘 1 件ごとに Haiku サブエージェントを並列起動**して confidence score をつける。

**Haiku への委譲プロンプト雛形:**

```
あなたはコードレビュー指摘の信頼度採点エージェントです。以下の指摘が実際のバグ・問題として妥当かを、0〜100 のスケールで採点してください。

指摘:
<summary + detail + file:line>

参考情報:
- design.md path: <path>
- impl.md path: <path>
- 該当ファイルの当該行付近: <抜粋>

採点スケール:
- 0: false positive。軽く精査すれば違うと分かる、または pre-existing。
- 25: 実際に問題か検証しきれない。可能性はあるがぼんやり。
- 50: 実際に問題だが nitpick 寄り、または実務上あまり起きない。
- 75: 実際に問題で、実務で当たる可能性が高い。design.md / impl.md 記載事項への違反ならここ以上。
- 100: 確実に問題で、頻繁に当たる。エビデンス（該当コード）が直接示せる。

出力: { "score": <0-100>, "reason": "<採点理由 1〜2 文>" }
```

**閾値:** 既定 **80**。80 未満はレポートに含めない。プロジェクトごとに厳しさを変えたい場合は本 skill のこの数値を上書きしてよい。

**note:** 「仕様に対する疑問（unresolved_questions）」は採点対象外。閾値でフィルタせず、そのままオーケストレーターに返す。

## モデル選択まとめ

| レンズ / ステップ | モデル |
|-----------------|-------|
| Lens 1: 外部仕様遵守 | Opus |
| Lens 2: 内部整合性 | Sonnet |
| Lens 3: バグ・エッジケース | Sonnet |
| Lens 4: デグレ影響 | Sonnet |
| Lens 5: 保守性・可読性 | Sonnet |
| Confidence scoring | Haiku |

「判断余地の大きさ」で決めるという subagent-orchestration の原則に従う。仕様との照合は判断が重いので Opus、他レンズは観点が絞られているので Sonnet、採点は機械的なので Haiku。

## 実行制約

- **並列度**: 5 レンズは同一ターンで並列委譲する（依存関係なし）
- **コンテキスト管理**: 各レンズも subagent-orchestration の 50% ルールに従う。自己申告 50% を超えた場合、そのレンズの成果と未完了範囲を明示して停止 → 必要ならオーケストレーターが新規サブエージェントに再委譲
- **タイムアウト**: レンズ 1 件が明らかに詰まった場合（応答不能）、そのレンズは skip して他レンズの結果でレポートを作る。skip したレンズはレポート先頭に明記する

## implement-review-loop への引き渡し

オーケストレーターは各レンズと Haiku の結果を集約し、implement-review-loop に以下の形で返す:

1. **未解決の疑問**（先頭）: 各レンズの `unresolved_questions` を統合したリスト。仕様に関わるものは implement-review-loop の分類フェーズで design.md 未決事項に上げる
2. **生き残った指摘**: `category_hint` で仕様・設計 / 実装判断に分けて並べる
3. **skip したレンズ**（あれば）: なぜ skip したかを明記
4. **各レンズのコンテキスト使用率**: 次回以降のレビューで新規サブエージェントに切り替える判断材料

implement-review-loop 側の step 4（指摘の分類）は、`category_hint` を初期値として使いつつ、最終判断はオーケストレーターが行う（ヒントはあくまでヒント）。

## 単独起動する場合

implement-review-loop の外から本 skill を呼ぶこともある（例: 「この差分だけ独立にレビューして」）。その場合も入力（design.md / impl.md / 差分）と出力形式は変えない。呼び元が implement-review-loop でないだけで、レビュー処理そのものの構造は同じ。
