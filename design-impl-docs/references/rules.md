# design.md / impl.md 運用規約 — ルール本体

## なぜこの運用か

仕様・要件と実装詳細を 2 つのファイルに分離して常に最新化しておくと:

- **セッションをまたげる** — 会話コンテキストが失われても、2 ファイルを読むだけで作業を再開できる
- **委譲が軽くなる** — サブエージェントにはファイルパスを渡すだけで必要なコンテキストを共有できる
- **決定が遡れる** — 仕様の決定履歴が 1 箇所に集まり、「なぜこうなっているか」を後から確認できる

「会話の中だけで決まったこと」はセッションが終われば消える。決定はファイルに置く。

## 配置

デフォルトは機能・タスク単位で `docs/<feature>/` 配下に置く:

```
docs/
└── <feature>/
    ├── design.md   # 仕様・要件（ユーザーと Claude が編集）
    └── impl.md     # 実装詳細（Claude のみ編集）
```

プロジェクトの CLAUDE.md に別の配置規約がある場合はそちらを優先する。

**作成タイミング:**

- **design.md** — タスク開始時に必ず作成する。仕様・要件が固まらないと以降のフェーズに進めないため、upfront に必須。
- **impl.md** — 実装フェーズに入る前に必ず作成する。実装エージェントへ委譲する前に、構成・技術的判断・実装ステップ・コミット分割案などの方針を impl.md 上で固めてから実装に着手する。方針固めなしに実装から入って後付けで impl.md を書くと、実装が終わってから設計を書き直すことになり、レビュー時のリファレンスにもならない。impl.md を書き始めた段階で design.md の穴に気づけばそこで手戻りが最小化できる、というのが upfront 作成の目的。

## design.md — 仕様・要件

**編集者: ユーザーと Claude の両方。レビュー: ユーザー。仕様の決定者は常にユーザー。**

仕様・要件に関する情報はすべてここに集約する。会話で決まった仕様も、必ず design.md に反映してから実装に進む。含める内容:

- **目的・背景** — なぜこの機能を作るか
- **要件** — 機能要件・非機能要件
- **仕様** — 外部から観測できる振る舞い（API 契約、UI 挙動、エラー時の動作など）
- **決定事項** — 決定した内容・日付・理由
- **未決事項** — ユーザーの判断待ちの項目

Claude が design.md に書くのは「ユーザーとの会話で決定した内容の記録」が中心。**ユーザーの決定なしに仕様を書き換えたり、未決事項を勝手に決定済みにしたりしない。** 仕様の空白に気づいたら、未決事項として記録した上でユーザーに確認する。

### 未決事項 → 決定事項の移動

ユーザーが判断を下した項目（design.md に直接書き込まれた場合を含む）は、Claude が責任を持って以下の整理を行う:

1. **未決事項から削除**する — 決定済みの項目を未決事項に残さない
2. **決定事項セクションに転記**する — 決定日・決定内容・理由（トレードオフのどれを選んだか）を含める。関連する既存の決定事項があれば、テーマごとにまとめて整理する
3. **決定事項を古いままにしない** — 後から仕様変更で覆された決定は「後継の決定事項に置き換え、旧内容は履歴として『変更履歴』サブセクションに残す」形で整理する（消さずに、なぜ変わったかを追えるようにする）

決定事項セクションは時系列の追記ログではなく「現在有効な仕様の決定集」として整理された状態を保つ。追記だけでゴチャつくと、後から読むサブエージェントが古い決定を拾って迷う。

## impl.md — 実装詳細

**編集者: Claude のみ。レビュー: ユーザー。** ユーザーは編集しないが、内容はレビューする。Claude が更新したら、どこを更新したかを報告してユーザーがレビューできる状態にする。

実装の現在地を示すドキュメント。含める内容:

- **構成** — 主要ファイル・モジュールの地図
- **技術的判断** — design.md の仕様をどう実現しているか、選択の理由
- **実装状況** — 完了 / 進行中 / 未着手
- **既知の制約・TODO**

「コードを読めばわかること」を書き写すのではなく、コードから読み取りにくい判断・経緯・全体像を書く。実装が進むたびに更新し、impl.md を読めば実装の現在地がわかる状態を保つ。

## 更新のタイミング

| イベント | すること |
|---------|---------|
| タスク開始 | design.md が無ければ作成。実装フェーズに入る前に impl.md も必ず作成し、構成・技術的判断・実装ステップ・コミット分割案を upfront に書く。既存タスクの再開時は design.md と impl.md を読んでから作業に入る |
| 仕様・要件の決定 | design.md の決定事項に転記して整理し、対応する未決事項を削除する（上記「未決事項 → 決定事項の移動」参照） |
| ユーザーが未決事項に判断を書き込んだ | 判断内容を決定事項に転記して整理し、未決事項から該当項目を削除してから作業を再開する |
| 実装の進行・技術判断 | impl.md を更新 |
| ユーザーが design.md を編集した | 差分を特定し、impl.md・実装・進行中タスクへの影響を反映 |

## 「design.md に追記したよ」と言われたら

1. design.md を読む
2. 前回把握していた内容との**差分**を特定する
3. 差分が実装・impl.md・進行中の作業に与える影響を洗い出す
4. 影響と対応方針をユーザーに報告してから作業に反映する

差分の解釈に迷ったら、推測で進めずユーザーに確認する。

## セッション再開時

新しいセッションで既存タスクに触れるときは、まず該当の design.md と impl.md を読んでから作業に入る。会話履歴や記憶だけで進めない。ドキュメントと実装が食い違っていたら、どちらが正か勝手に判断せずユーザーに確認する。
