# subagent-orchestration スキル with_skill / without_skill 比較 — iteration 1

**Verdict:** ベースライン（スキルなし）は「サブエージェントを起こす」までは自発的にやるが、本スキル固有の 5 系統の規約（規模を問わない委譲 / 曖昧仕様の design.md エスカレーション / 調査 = Opus / 委譲プロンプトの design.md/impl.md 参照 / コンテキスト自己申告）はベースラインでは発現しない。

| | with_skill | without_skill | Δ |
|---|---|---|---|
| Assertion pass | **20 / 20 (100%)** | 11 / 20 (55%) | **+45pt** |

## 方法

- **試行**: 5 evals × 2 configs = 10 runs
- **被験モデル**: Claude Opus 4.7（session default）
- **手順**: サブエージェントに同じユーザープロンプトを渡し、`with_skill` 側だけ `SKILL.md` と `rules.md` を先に読ませる。両方とも「実装は行わずプラン記述のみ」を指示し、末尾に構造化 JSON を吐かせて機械採点。
- **観測点**: サブエージェントを起こしたか、だけでなく本スキル固有の以下 5 系統に置いた。
  1. 規模を問わず委譲するか
  2. 曖昧仕様を design.md 未決事項セクション経由でエスカレーションするか
  3. 調査 = Opus / 実装 = Sonnet と使い分けるか
  4. 委譲プロンプトに design.md / impl.md のパスと「まず読め」を含めるか
  5. コンテキスト使用率の自己申告指示を含めるか

## 解釈

### スキルが確実に必要な 3 領域

- **A. 小規模タスクでも委譲する規約** — without は「数行の変更だから自分で編集する」と判断。オーケストレーターのコンテキスト保護は規模で例外を許すと崩れる。
- **B. 曖昧仕様の design.md 経由エスカレーション** — without は汎用スキーマで一般化して勝手に決める傾向。「仕様の決定者は常にユーザー」は skill 固有。
- **C. design.md/impl.md がプロンプトに書かれていない状況での委譲プロンプト設計** — without は調査委譲プロンプトに design.md/impl.md 参照を入れず、Opus 指定もしない。

### スキルなしでも再現する挙動

- **D. 独立タスクの並列委譲** — 両条件 pass。Claude Code の system prompt に既に組み込まれている。
- **E. プロンプトに design.md/impl.md が明示された実装委譲** — 両条件満点。文脈が明示されていれば自発的に構造化される。

→ **改善方向**: 並列化・エスカレーション境界の一般記述はコンパクトに保ち、eval-1 / eval-2 の規範（規模を問わない委譲・design.md 経由エスカレーション）にウェイトを寄せる余地あり。

## スキルで何ができるようになるか

数値上の pass 率の差だけではなく、このスキルが導入する**状態の外部化**（design.md / impl.md が真の source、CC セッションのメモリは source ではない）は、ユーザー側の作業パターンを次のように広げる。

### 01. 複数 backlog item の並列進行

現状はコンテキスト切り替えのコストが高いため、item を 1 つずつ進める運用。導入後: item ごとに design.md/impl.md が独立して状態を持つため、セッション A で item-42、セッション B で item-43 を同時に走らせられる。オーケストレーターは thin なので、ユーザーには「決めるべきこと」だけが浮かんでくる。CC 側の文脈リロードは不要（docs から復元できる）。

### 02. 決定 / 実装 / レビューの階層分離

決めるべきことは design.md、進捗と実装詳細は impl.md、コード変更はサブエージェントが返す diff。ユーザーは「決定タスク」だけをこなせばよく、実装ログを追う必要がない。レビュー負荷が減り、判断の質に集中できる。

### 03. セッション事故に強い長期作業

サブエージェントの寿命が短く（50% 閾値で切り替え）、都度成果を design.md/impl.md に落とすため、長期作業がセッションの compact / crash / 誤 clear で消えない。翌週再開時に「あの続き」を思い出す必要もない。

### 04. 未決事項ドリブンな中断・再開

仕様の疑問が出た時点で design.md に「未決事項」として書き出して item を止められる。ユーザーはミーティング・ヒアリング等で決めてから戻り、決定事項を design.md に転記して再開する。未決事項の存在自体が文書化されているので忘れず、item が塩漬けにならない。

### 05. 引き継ぎ可能な状態

design.md/impl.md を読めば別のセッション・別の CC ユーザー・別チームメンバーが状態を把握できる。「私の CC セッションを見てくれ」ではなく「docs を見てくれ」で済む。将来的な自動化（cron / 夜間ジョブ）や別 AI への引き継ぎにも同じ経路が使える。

### 典型的なワークデイの変化

| 局面 | 現状 | スキル導入後 |
|---|---|---|
| 仕様の空白に当たる | その場でチャットで即席の判断を求められる。決めきれないと item が塩漬け。 | サブエージェントが停止 → 選択肢と推奨を design.md 未決事項に追記 → 別 item に切り替え。 |
| 並列作業 | コンテキスト切り替えを避けるため 1 item 集中。 | 複数セッションで並列進行。状態は docs にあるので、ユーザー側の切り替えが軽い。 |
| 週をまたぐ作業 | セッションの寿命に頼るため、compact / crash で消える。 | design.md/impl.md から新セッションが再起動。作業の連続性がセッション寿命から独立。 |
| ユーザーの認知負荷 | 実装詳細・調査ログもチャットに流れ、判断ノイズが混ざる。 | メイン CC は意思決定と指揮のみ。ユーザーは決定タスクだけを見る。 |

## 結論

スキル固有の 3 系統の規範（規模を問わない委譲 / 曖昧仕様の design.md 経由エスカレーション / 委譲プロンプトの design.md 参照とモデル選択）はベースラインでは再現しない。加えて、状態を design.md/impl.md に外部化する構造は上記のワークデイの変化を可能にする。**本スキルの価値は「委譲する / しない」の二値だけではなく、ユーザーの作業パターンそのものを広げる点にある。**

ただし eval-4 / eval-5 の結果から、スキル本体には Claude Code のデフォルト挙動と重複する記述もある。次の改善として、並列化・impl.md 更新義務・エスカレーション境界の一般記述を短縮し、逆に **eval-1 / eval-2 に関わる規範（規模を問わない委譲・design.md 経由エスカレーション）**にウェイトを寄せることを検討する価値がある。

## 注意点

1. **N=1**。特に境界近い assertion は再試行でぶれる可能性。信頼区間が欲しければ 3 seeds × 5 evals × 2 configs = 30 runs にスケール可能。
2. **「プラン記述」評価であり実行評価ではない**。モデル自身に「あなたはどう動きますか」を答えさせている。実挙動を測るなら実際の Task 呼び出し trace 分析が必要。
3. **被験は Opus 4.7 のみ**。Sonnet 4.7 / Haiku では without 側の pass 率が下がる可能性が高い（＝スキルの必要性はさらに上がる方向）。定期再測を推奨。
4. **description ベースの自動 trigger は未検証**。skill-creator の別ループで検証可能。
