# プラン: Next.js 認証周りの調査とリファクタ候補洗い出し

## 冒頭プラン（自然文）

このタスクは「調査」に分類されるため、subagent-orchestration の規約に従い、自分では読み込まず調査サブエージェントに委譲します。まずオーケストレーターとしてリポジトリ直下に `design.md` / `impl.md` が存在するか（および認証機能に対応するセクションがあるか）だけを軽く確認し、無ければ調査結果を受け取った後にドキュメントエージェントで新規作成する段取りにします。調査自体は Opus モデルの調査エージェント 1 本に委譲します。認証周りは「サーバー側セッション/ミドルウェア」「クライアント側 UI・状態」「設定・依存」など観点が絡み合っており、全体像を 1 人が持つ方がリファクタポイントの評価精度が高いため、今回は並列化しません（規模が想定以上に大きいと分かった段階で観点別に切り分ける）。委譲プロンプトには、役割・design.md/impl.md のパスと「まず読め」の指示・タスクと成果物形式・エスカレーション指示・境界・コンテキスト使用率 50% での自己申告と停止規約を必ず入れます。仕様不明点（例: 認証プロバイダー移行の可否、セッション戦略の変更、権限モデルの見直しなど）は調査エージェントに判断させず返させ、選択肢と推奨を付けて design.md の「未決事項」セクションに追記した上でユーザー確認に回します。ユーザーへの最終報告時は、規約通りエスカレーション事項を先頭に、次いで「現状構造の要約」「リファクタ候補（優先度と根拠）」「関連ファイルパス」の順で提示します。

# コンテキスト
- 対象: 本リポジトリの認証機能全般（ログイン/ログアウト、セッション、ミドルウェア、認可、外部プロバイダー連携、環境変数、UI）
- design.md: <リポジトリ直下 design.md のパス。無ければ「未整備」と報告してよい>
- impl.md: <リポジトリ直下 impl.md のパス。無ければ「未整備」と報告してよい>
- 最初に上記 2 ファイルを Read で読み、既存の仕様・実装記録を把握してから調査に入ってください。

# タスク
1. 認証に関わるファイルを網羅的に洗い出す（app/ or pages/ 配下の route handlers / server actions / middleware.ts / lib や services 配下の auth 系モジュール / hooks / context / UI コンポーネント / 型定義 / 環境変数の参照箇所 / package.json の依存）。
2. 現状の構造を「サーバーサイド」「クライアントサイド」「共有（型・ユーティリティ）」「設定・依存」の 4 レイヤに分けて要約する。認証方式（next-auth / Auth.js / Clerk / 自作 JWT / Cookie セッション など）と、セッション保存場所・保護されているルートの判定方法を明記する。
3. リファクタしやすいポイントを「変更容易性 × 影響範囲 × リスク」で 3〜7 個に絞って提示する。各項目に (a) 該当ファイル・行, (b) 現状の何が問題か, (c) 推奨する方向性, (d) 触るべきでない副作用境界 を書く。
4. 仕様・要件レベルで判断が必要な論点（例: セッション戦略の切り替え可否、権限モデルの再設計、外部プロバイダー変更）は自分で決めず、「未決事項」として選択肢とトレードオフを添えて返してください。

# 成果物フォーマット
- 関連ファイルパス一覧（絶対パスまたはリポジトリルート相対）
- 現状構造の要約（上記 4 レイヤ）
- リファクタ候補リスト（優先度付き）
- 未決事項リスト（あれば）
- 自身のコンテキスト使用率（末尾、概算 %）

# 境界
- コード・ドキュメントを一切書き換えない（Read/Grep/Glob のみ）
- タスク範囲外（テスト追加、最適化、依存更新など）に手を出さない
- 迷ったらエスカレーション（未決事項として返す）に倒す

# コンテキスト管理
- 使用率が 50% に近づいたら、そこまでの成果と未完了リストを明文化して停止してください。続きは別エージェントに引き継ぎます。

## 想定フロー

1. オーケストレーターが design.md / impl.md の存在を Read で確認
2. 上記プロンプトで調査エージェント（Opus）1 本を起動
3. 成果物を受領 → 未決事項があれば design.md に「未決事項」セクションを（ドキュメントエージェント経由で）追記し、ユーザーに「design.md を見て決定してください」と依頼
4. 決定後、リファクタ実行フェーズは別途「実装エージェント（Sonnet）」に委譲（今回のプロンプトの範囲外）
