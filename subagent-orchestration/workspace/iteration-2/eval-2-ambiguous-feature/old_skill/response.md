# プラン: ユーザープロフィール画面へのバッジ機能追加

「バッジ」の仕様が全く決まっていない（種類・取得元・付与条件・表示形式・件数上限・並び順など何一つ確定していない）ため、いきなり実装に入るのは規約違反です。私自身の判断で仕様を決めることもしません。まず現状把握を委譲し、その結果をもとに design.md の「未決事項」を整理してユーザーに決定を仰ぎ、決定後に実装を委譲します。

このターンでは**調査エージェント（Opus）1 つ**だけを起動します。目的は 2 つ：(a) `docs/user-profile/` の design.md / impl.md の有無確認と、なければ骨子作成、(b) 既存プロフィール画面まわりの構造把握（コンポーネント配置・データ取得経路・スタイル体系）。調査結果を受け取ったら、私の側で design.md の「未決事項」セクションに「バッジとは何か（実績系？ロール系？イベント系？）」「取得元 API」「表示位置・形状・件数上限」「クリック時の挙動」などを選択肢と推奨付きで書き出し、ユーザーに「design.md の未決事項セクションを見て決定してください」とだけ依頼します。チャットに選択肢を長々貼らず、質問は design.md 経由。決定後、初めて実装エージェント（Sonnet）に委譲し、impl.md 更新まで指示します。

## 調査エージェントへの委譲プロンプト（実際に渡す本文）

```
あなたはコードベース調査の専門エージェントです。

# コンテキスト
- 対象機能: ユーザープロフィール画面へのバッジ表示追加
- 参照ドキュメント: docs/user-profile/design.md, docs/user-profile/impl.md
  （どちらも未作成の可能性がある。存在確認から始め、なければ骨子だけ作成して構わない）

# タスク
1. docs/user-profile/design.md / impl.md の存在を確認し、なければ空の骨子（見出しのみ）を作成する
2. 既存のユーザープロフィール画面の実装場所・コンポーネント構造・データ取得層（API クライアント / hooks / store）・スタイル体系（デザインシステム、Tailwind か CSS Modules かなど）を洗い出す
3. 「バッジ」に類する既存概念（アイコン、ラベル、称号、実績、ロールタグ等）がリポジトリ内に既にあるか grep して確認する
4. バッジ機能を追加する場合に影響しそうなファイル・API・型定義の候補を列挙する

# 成果物
- 関連ファイルパスの一覧と各役割
- 現状のプロフィール画面の構造要約
- 既存のバッジ類似実装の有無と、あれば再利用可否の所見
- 仕様上の未解決の疑問（バッジの意味・取得元・表示形式・件数上限・クリック挙動など、design.md に書くべき論点）を「疑問 / 選択肢 / 各選択肢のトレードオフ / 推奨」の形で列挙

# 境界
- コードの変更はしない（骨子ドキュメント作成のみ可）
- 仕様に関わる疑問は自分で判断せず「未解決の疑問」として返す
- タスク範囲外の調査（他機能）はしない

# コンテキスト管理
- 成果物の末尾に自身のコンテキスト使用率（概算）を必ず含めること
- 使用率が 50% に近づいたら、そこまでの成果と未完了リストを明文化して停止すること
```

## 調査完了後の流れ（このターンでは実行しない）

1. 返ってきた「未解決の疑問」を私が design.md の「未決事項」セクションに整理して追記
2. ユーザーに「design.md の未決事項を見て決定してください」と口頭で依頼（選択肢はチャットに貼らない）
3. 決定後、design.md の「決定事項」へ移動
4. 実装エージェント（Sonnet）に、design.md / impl.md のパス・エスカレーション指示・境界・コンテキスト自己申告・impl.md 更新義務を含めた委譲プロンプトで実装依頼
5. 完了後にレビューエージェント（Opus）で仕様乖離チェック、ユーザーへ要約報告（エスカレーションがあれば先頭に）
