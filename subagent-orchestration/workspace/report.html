解釈

スキルが確実に必要な 3 領域

  • A

    小規模タスクでも委譲する規約

    without では「数行の変更だから自分で編集する」と判断される。オーケストレーターのコンテキストを対話に集中させる目的は、規模で例外を許すと崩れる。

  • B

    曖昧仕様の design.md 経由エスカレーション

    without では汎用スキーマで一般化して勝手に決める傾向が強い。「仕様の決定者は常にユーザー」という規範は skill 固有。

  • C

    ユーザーが design.md/impl.md を明示していない状況での委譲プロンプト設計

    without では調査を委譲するときの委譲プロンプトに design.md/impl.md への参照は入らず、モデル指定も Sonnet に流れる。文脈を委譲プロンプトで再確立するのは skill 固有。

スキルなしでも再現する挙動

  • D

    独立タスクの並列委譲 — eval-5 で両条件とも pass

    Claude Code の system prompt に既に「独立タスクは並列」が入っており、skill で重ねて指示する必要は薄い。skill 内での記述はコンパクトに保ってよい。

  • E

    プロンプトに design.md/impl.md パスが書かれた実装委譲 — eval-4 で両条件とも満点

    ユーザーが文脈を明示していれば、モデル選択・impl.md 更新指示・エスカレーション境界は自発的に出る。skill が価値を出すのは「文脈がプロンプトに書かれていないケース」。

スキルで何ができるようになるか

数値上の pass 率の差だけではなく、このスキルが導入する状態の外部化（design.md / impl.md が真の source、CC セッションのメモリは source ではない）は、ユーザー側の作業パターンを次のように広げる。

  • 01

    複数 backlog item の並列進行

    現状はコンテキスト切り替えのコストが高いため、item を 1 つずつ進める運用。
    導入後: item ごとに design.md/impl.md が独立して状態を持つため、セッション A で item-42、セッション B で item-43 を同時に走らせられる。オーケストレーターは thin なので、ユーザーには「決めるべきこと」だけが浮かんでくる。CC 側の文脈リロードは不要（docs から復元できる）。

  • 02

    決定 / 実装 / レビューの階層分離

    決めるべきことは design.md、進捗と実装詳細は impl.md、コード変更はサブエージェントが返す diff。ユーザーは「決定タスク」だけをこなせばよく、実装ログを追う必要がない。レビュー負荷が減り、判断の質に集中できる。

  • 03

    セッション事故に強い長期作業

    サブエージェントの寿命が短く（50% 閾値で切り替え）、都度成果を design.md/impl.md に落とすため、長期作業がセッションの compact / crash / 誤 clear で消えない。翌週再開時に「あの続き」を思い出す必要もない。

  • 04

    未決事項ドリブンな中断・再開

    仕様の疑問が出た時点で design.md に「未決事項」として書き出して item を止められる。ユーザーはミーティング・ヒアリング等で決めてから戻り、決定事項を design.md に転記して再開する。未決事項の存在自体が文書化されているので忘れず、item が塩漬けにならない。

  • 05

    引き継ぎ可能な状態

    design.md/impl.md を読めば別のセッション・別の CC ユーザー・別チームメンバーが状態を把握できる。「私の CC セッションを見てくれ」ではなく「docs を見てくれ」で済む。将来的な自動化（cron / 夜間ジョブ）や別 AI への引き継ぎにも同じ経路が使える。

結論

スキル固有の 3 系統の規範（規模を問わない委譲 / 曖昧仕様の design.md 経由エスカレーション / 委譲プロンプトの design.md 参照とモデル選択）はベースラインでは再現しない。加えて、状態を design.md/impl.md に外部化する構造は、上記のワークデイの変化を可能にする。本スキルの価値は「委譲する / しない」の二値だけではなく、ユーザーの作業パターンそのものを広げる点にある。

iteration-2 では、iteration-1 で示唆した短縮方向（並列化セクション廃止・具体例削除・重複記述削減）を実際に適用しても pass 率が 100% 保持されることを確認した。skill は必要かつ運用可能なサイズに収まる。次のイテレーションで検討する余地: (a) SKILL.md 本体の圧縮、(b) description ベースの自動 trigger 精度検証、(c) 弱いモデル層 (Sonnet 4.7 / Haiku 4.5) での効き目測定。

注意点

  • 1

    各 iteration で N=1 の測定

    各条件 1 seed のみ。iteration-1 で eval-1 / eval-2 は質的に非対称な差 (with 100% / without 0%) が観測されたためノイズではないと判断できるが、境界近い assertion（例: eval-3 の Opus 選択）は再試行でぶれる可能性あり。信頼区間が欲しければ 3 seeds × 5 evals × 2 configs = 30 runs にスケールできる。

  • 2

    「プラン記述」評価であり実行評価ではない

    実際にサブエージェントを起こしたわけではなく、モデル自身に「あなたはどう動きますか」を答えさせている。実行時挙動と乖離する余地がある（一般に、モデルは「やる」と言って「やらない」ことがある）。実挙動を測るなら、実際に Task ツール呼び出しを許可した trace の分析が必要。

  • 3

    被験モデルは Opus 4.7 のみ

    Sonnet 4.7 や Haiku 4.5 では without 側の pass 率が下がる可能性が高い（＝スキルの必要性はさらに上がる方向）。逆に将来のより強いモデルでは without 側が改善するかもしれない。定期再測を推奨。

  • 4

    with_skill でスキルを「必ず読ませた」プロンプト構造

    実運用では description ベースの自動 trigger が入る。trigger 精度は本測定では検証していない（skill-creator の別ループで検証可）。
