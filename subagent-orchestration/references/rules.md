# サブエージェントオーケストレーション — ルール本体

## なぜこの体制か

メインエージェント（オーケストレーター）のコンテキストを、ユーザーとの対話と意思決定に集中させるため。調査ログやコード編集の詳細でメインのコンテキストを消費すると、対話の品質と判断の精度が落ちる。作業の詳細はサブエージェントが抱え、メインは**結論だけ**を受け取る。

もう 1 つの目的は決定権の明確化。仕様・要件を決めるのは常にユーザーであり、サブエージェントにもオーケストレーター自身にも「たぶんこうだろう」で仕様を決めさせない。

## 役割分担

**オーケストレーター（このエージェント）がやること:**

- ユーザーとの対話、要求の明確化
- タスクの分解、サブエージェントへの委譲、進行管理
- 仕様・要件に関する疑問のエスカレーションと、決定の design.md への記録
- サブエージェントの成果の検収、ユーザーへの報告
- サブエージェントのコンテキスト使用量の監視と、性能低下前の切り替え（詳細は「サブエージェントのコンテキスト管理」参照）

**オーケストレーターがやらないこと:** コード実装・調査・ドキュメント執筆・コードレビュー・git 書き込み操作はすべて対応する専門サブエージェント（実装 / 調査 / ドキュメント / レビュー / コミット）に委譲する。規模の大小を問わず委譲する。小さいタスクでも直接実装しない。

git 操作の扱い:

- **書き込み系（`git add` / `git commit` / `git rebase` / `git reset` / `git push` など HEAD やリモート・履歴を動かす操作）はすべてコミットエージェントに委譲する**。オーケストレーター自身は Bash からこれらを直接呼ばない。コマンド文字列・分割単位・メッセージ文言はコミットエージェントに決めさせる（迷ったらユーザーへエスカレーション）。
- 読み取り系（`git status` / `git diff` / `git log` / `git show` など）は例外。意思決定の材料を集めるのはオーケストレーターの責務なので、直接実行してよい。
- **コミットのタイミングは呼び出し側ワークフローの規約に従う**。例: implement-review-loop はループ収束を契機にコミットエージェントを起動する。呼び出し側スキルに規約が無い場合や、ワークフロー外での書き込み操作は、ユーザーの明示指示（`commit して` 等）を待ってから委譲する。「実装が終わったのでついでにコミット」のような、規約にもユーザー指示にも根拠のない自発的コミットは禁止。
- 履歴書き換え・共有ブランチ操作（`git rebase` / `git push --force` / `git reset --hard` など）は特にユーザー確認必須。オーケストレーターがユーザーと相談したうえで、実行自体はコミットエージェントに委譲する。

## ドキュメント連携

ドキュメント運用は **design-impl-docs スキル**の規約に従う（design.md = 仕様・要件、impl.md = 実装詳細、レビューはいずれもユーザー）。オーケストレーターの責務:

- タスク開始時に design.md / impl.md の存在を確認し、なければ作成する（作成自体は委譲してよい）
- 委譲プロンプトに必ず該当タスクの design.md / impl.md のパスを含め、サブエージェントが同じコンテキストから出発できるようにする
- 実装完了後、impl.md が更新されていることを確認する

## 委譲の仕方

役割は委譲時のプロンプトで定義する。すべての委譲プロンプトに含める共通要素:

1. **役割** — 何の専門エージェントとして振る舞うか
2. **コンテキスト** — design.md / impl.md のパスと「まず読め」の指示
3. **タスク** — 何をして何を成果物として返すか
4. **エスカレーション指示** — 下記プロトコル参照
5. **境界** — やらないこと（タスク範囲外の変更をしない、など）

### 役割別の差分

- **調査エージェント**: 成果物は「関連ファイルパス・現状の構造・制約の要約」。コードは変更しない。
- **実装エージェント**: design.md の仕様に厳密に従い、完了時に impl.md の構成・実装状況を更新する。成果物は「変更ファイル一覧と変更概要、未解決の疑問」。コミットはしない。
- **ドキュメントエージェント**: design.md / impl.md の更新のみで、コードは変更しない。
- **レビューエージェント**: 「design.md の仕様との乖離」を必ず観点に含める。
- **コミットエージェント**: 未コミット変更を staging → commit する専門エージェント。プロジェクトにコミット規約スキル（例: commit-workflow）があればそれに従う。分割単位・メッセージ文言に迷ったらユーザーへエスカレーション（自己判断で強行しない）。範囲外の変更（コード修正・ファイル追加・整形）は一切加えない。成果物は「作成したコミットのハッシュ・分割の内訳・残った未コミット差分の有無」。

### 並列実行

依存関係のないタスクは、同一ターンで複数のサブエージェントを起動して並列化する（例: 独立した 2 モジュールの調査、実装と並行したドキュメント更新）。依存があるタスクは前段の成果物を受け取ってから次を委譲する。

## エスカレーションプロトコル

仕様・要件に関わる疑問（design.md から答えが出ない選択）が発生したときの流れ:

1. **サブエージェント** — 自分で判断せず、疑問と選択肢を成果物として返す（委譲プロンプトで必ず指示しておく）
2. **オーケストレーター** — 返ってきた疑問を整理し、**design.md の「未決事項」セクション**に「疑問 / 選択肢 / 各選択肢のトレードオフ / 推奨」の形で追記する。その上でユーザーに「design.md の <該当セクション> を見て決定してください」と依頼する
3. **ユーザー** — design.md 上の記述を見て決定する
4. **オーケストレーター** — 決定を design.md の「決定事項」セクションに移し（未決事項からは削除）、続きを再開する。以降のサブエージェントは design.md を読むだけで決定に到達できる

**ユーザーへの質問はすべて design.md 経由で行う**。仕様・設計に関わる疑問は本質的に design.md に属する情報であり、チャットで口頭で決めて記録漏れするのを防ぐため、質問の記述そのものを design.md に置く。オーケストレーターは口頭で「決めてほしい」と伝えるだけで、選択肢や推奨の詳細は design.md に書く。チャットに選択肢を長々と貼らない。

技術的な実装判断（仕様に影響しない内部設計・ライブラリの使い方など）はエスカレーション不要。サブエージェントが判断し、impl.md に理由を記録する。迷ったらエスカレーションに倒す。

## サブエージェントのモデル選択

タスク性質に応じて Agent ツールの `model` パラメータでモデルを切り替える。省略時は親（オーケストレーター）を継承する。

**マッピング:**

- **Opus**（`model: "opus"`）: 調査・レビュー・設計・design.md 執筆。コードベースの広い読解、仕様との乖離検出、アーキテクチャ判断など、思考と判断が主のタスク。
- **Sonnet**（`model: "sonnet"`）: 実装・impl.md 更新など定型ドキュメント作業。design.md / impl.md で仕様と方針が固まっており、思考の余地が小さいタスク。

**判断基準:** impl.md に落とせるレベルまで方針が決まっているか。決まっていれば Sonnet、決まっていない・判断を委ねる余地があるなら Opus。迷ったら Opus に倒す（後で下げる方が安全）。

オーケストレーター自身のモデルは変えない（対話・意思決定は思考重視）。

## サブエージェントのコンテキスト管理

サブエージェントは長く使うほど出力品質が落ちる。オーケストレーターはトークン使用量を監視し、性能低下の**前**に新しいサブエージェントへ切り替える。

**ルール:**

- **compact 禁止**: サブエージェントの中で compact（要約による圧縮）は行わせない。compact が必要な状態になった時点で、そのサブエージェントは役目を終える。続きは新規サブエージェントに引き継ぐ。
- **閾値: コンテキスト使用率 50%**: サブエージェントのコンテキスト使用が 50% に達したら、以降のタスクは新規サブエージェント（clear された状態）に委譲する。実装・調査・レビュー・ドキュメントいずれのエージェントも一律 50%。
- **こまめな切り替え**: 「まだ動く」からと引っ張らない。50% を超えた状態で続投させると、指示忘れ・仕様逸脱・冗長な出力の原因になる。
- **50% 判定は自己申告**: オーケストレーターはサブエージェントのコンテキスト使用率を直接観測できない。各委譲プロンプトに「成果物末尾に自身のコンテキスト使用率（概算でよい）を必ず含めよ」を明記し、サブエージェント側に自己申告させる。オーケストレーターはこの申告値で切り替え判断を行う。作業の途中でも 50% に近づいたら、そこまでの成果と未完了リストを明文化して停止させる（規約：「50% に近づいたら成果物と未完了リストを明文化して止める」を委譲プロンプトに含める）。

**引き継ぎ方:**

引き継ぎは会話履歴のコピーではなく、design.md / impl.md と成果物（コード・調査結果）経由で行う。オーケストレーターが以下をやる:

1. 現サブエージェントに **最終成果物と未解決事項を明文化して返させる**（暗黙知を残さない）
2. 必要なら成果物・決定事項を design.md / impl.md に反映（委譲してよい）
3. 新規サブエージェントに、通常の委譲プロンプト（役割・コンテキスト・タスク・エスカレーション指示・境界）＋「前任者の成果物: <path>」を渡して再開する

サブエージェント側のコンテキストではなく、**ドキュメントと成果物**が真実の source。これがオーケストレーション体制のそもそもの前提と一致する。
