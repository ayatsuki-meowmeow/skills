# コミット作成ワークフロー

## 基本方針

コミットは **意味単位** で分割する。1つのコミットには1つの意味単位だけを含める。
複数の意味単位の変更が混在している場合は、ファイル単位またはステージング操作で分けて、複数のコミットを作成する。

これは以下の理由による:

- **履歴の追跡しやすさ** — 「いつ・どの目的で何が変わったか」がコミット単位で読み取れる
- **レビューの容易さ** — レビュアーはコミット単位で意図を理解できる
- **リバートの安全性** — 不具合発生時に、影響範囲を最小化して戻せる

## 意味単位の分類

代表的な意味単位は以下のとおり。プロジェクトの構造に応じて適宜判断する。

| 意味単位 | 該当する変更の例 |
|---------|----------------|
| ビジネスロジック実装 | ドメインロジック、ユースケース、サービス層、APIエンドポイント |
| ユーティリティ実装 | 汎用ヘルパー、共通関数、型定義、定数 |
| UI 実装 | コンポーネント、ページ、スタイル、レイアウト |
| テスト | ユニットテスト、E2E テスト、テストヘルパー、フィクスチャ |
| ドキュメント | README、設計ドキュメント、コードコメントの大幅な追加 |
| 設定 | 設定ファイル、CI/CD、依存関係、Lint / Formatter 設定 |
| リファクタリング | 振る舞いを変えない構造変更（命名、抽出、分割） |
| バグ修正 | 振る舞いを変える修正（テスト追加と同時にコミットしてよい） |

ビジネスロジックと、それに対するテストを **同一コミットに含めるかは判断による**。原則として:

- 機能追加に対するテストはセットでコミットして良い（実装とテストの結びつきが明示される）
- 既存機能のテストを後から書き足す場合は分ける（テスト追加という単独の意味単位）

## コミットメッセージのフォーマット

```
<英語の1行サマリ>

<日本語による詳細説明>
```

- **1行目（タイトル）**: 英語で記述する。GitHub や CLI で目につく場所なので、変更の概要を簡潔に表現する。50〜72 文字を目安。末尾にピリオドは付けない
- **空行**: 1行目と本文の間には必ず空行を1行入れる
- **本文（詳細）**: 日本語で「何をしているか」を記述する。複数行でも箇条書きでも可。読み手が "なぜこのコミットが必要か" を理解できる程度の情報を含める

### 1行目の英語サマリの書き方

- 動詞の原形で始める（命令形）: `add`, `implement`, `fix`, `refactor`, `remove`, `update`, `rename`
- Conventional Commits（`feat:` / `fix:` 等のプレフィックス）はプロジェクトで採用していれば従う。指定がなければ素のサマリで良い
- 主語は省略する（"This commit adds..." のような書き方はしない）

### 例

```
implement user authentication service

ユーザー認証のビジネスロジックを実装。
- パスワードハッシュ化（bcrypt）
- セッショントークン生成
- ログイン試行回数の制限
```

```
add date formatting utility

日付フォーマット用のユーティリティ関数を追加。
ISO 8601 とローカルタイムゾーンの双方に対応する。
```

```
add tests for user authentication service

UserAuthService に対するユニットテストを追加。
- 正常系: ログイン成功・トークン生成
- 異常系: パスワード不一致・試行回数超過
```

```
fix off-by-one error in pagination

ページネーション計算でオフバイワンになっていた箇所を修正。
最終ページの件数が常に1件少なくなる問題を解消。
```

## コミッターの扱い

- コミッターは **依頼者本人のみ** とする
- `Co-Authored-By:` フッターは **付けない**
- Claude / AI ツールを示すフッターやサインも **付けない**
- `git commit` 実行時に user.name / user.email を変更してはいけない

これは依頼者本人がコードのオーナーシップを持つことを明示するためのルール。

## 実行手順

1. **状態確認**: `git status` と `git diff`（必要に応じて `git diff --staged`）で変更内容を把握する
2. **意味単位への分類**: ファイル一覧を眺め、どのファイルがどの意味単位に属するかを整理する
3. **ステージング**: 1つ目の意味単位に属するファイルだけを `git add <files...>` でステージング
   - 同一ファイル内に複数の意味単位が混在する場合は `git add -p` でハンク単位で分ける
4. **コミット作成**: 後述の HEREDOC 形式で複数行メッセージを書いてコミット
5. **次の意味単位へ**: 残りの変更について 3〜4 を繰り返す
6. **最終確認**: `git status` で未コミットの変更が残っていないことを確認する

### 複数行コミットメッセージの書き方

メッセージ内に空行や日本語が含まれるため、HEREDOC を使うのが安全:

```bash
git commit -m "$(cat <<'EOF'
implement user authentication service

ユーザー認証のビジネスロジックを実装。
- パスワードハッシュ化（bcrypt）
- セッショントークン生成
EOF
)"
```

`-m "..."` をシェルに直書きすると改行のエスケープ事故が起きやすいため、原則 HEREDOC を使う。

### ステージング時の注意

- `git add -A` / `git add .` は **使わない**。無差別に全ファイルをステージングすると意味単位での分割が崩れる
- 機密ファイル（`.env`, credentials など）が紛れ込んでいないか、ステージング前に確認する
- ファイル名を明示して `git add path/to/file1 path/to/file2` のように指定する

## やってはいけないこと

- 複数の意味単位を1つのコミットに混ぜる
- コミットメッセージの1行目を日本語にする（GitHub 等で読みづらくなる）
- `Co-Authored-By:` などのフッターを付ける
- `git add -A` / `git add .` で無差別ステージング
- `git commit --amend` を独断で使う（過去のコミットを書き換える操作なので明示依頼があるまで避ける）
- `--no-verify` でフック（pre-commit など）をスキップする（フック失敗時は原因を調査して直す）
- ユーザーが明示的に許可した場合を除き `git push` まで実行する

## 判断に迷ったら

意味単位の分け方が曖昧（例: 1つのファイルでロジックと UI を同時に変更している、リファクタリングと機能追加が混ざっている）な場合:

1. 分け方の案を提示する（A案: ファイル単位で分割 / B案: 1コミットにまとめる など）
2. 依頼者に判断を仰ぐ
3. 勝手に「えいやでまとめる」「無理やり分ける」をしない
