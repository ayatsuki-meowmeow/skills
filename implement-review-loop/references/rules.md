# 実装 → レビュー反復ループ — ルール本体

## なぜこのループか

design.md / impl.md が揃っても、一発の実装で仕様に完全準拠し、かつ品質的に問題のないコードが出てくることはまずない。実装エージェントは自分の書いたコードを客観視できないので、独立したレビューエージェントに指摘させ、指摘を潰す反復を回す。

反復を明文化しておかないと、こうなりがち:

- レビュー 1 周で満足して指摘を残したまま「完了」と報告する
- レビュー指摘の中で仕様に関わるものまで Claude が勝手に判断して黙って直す（ユーザーの意思決定を奪う）
- 指摘の履歴が残らず、後から「なぜこの実装になったか」を辿れない

このループの目的は、**レビュー指摘 0 件かつユーザーが動作確認できる状態**をゴールに、指摘の分類・エスカレーション・記録・対応を毎周同じ形で回すこと。

## 前提

- 該当タスクの `design.md` と `impl.md` が存在する（なければ design-impl-docs スキルに従って先に作る）
- サブエージェント（Task / Agent ツール）が利用できる
- 本ループは subagent-orchestration スキルの下位ワークフロー。役割分担・委譲プロンプトの共通要素・エスカレーションプロトコル・モデル選択・コンテキスト管理などは subagent-orchestration の rules.md に従う

## ループの全体像

```
[開始] design.md / impl.md 確認
   ↓
1. 実装エージェントに委譲（今回対応する範囲）
   ↓
2. レビューエージェントに委譲（実装差分をレビュー）
   ↓
3. 指摘を分類
   - 仕様・設計に関わる指摘 → ユーザーへエスカレーション（design.md 経由）
   - それ以外 → Claude が対応方針を決定
   ↓
4. design.md / impl.md に「レビュー指摘と対応」を記録
   ↓
5. 指摘 0 件？
   - No → 対応を実装エージェントに委譲して 2 に戻る
   - Yes → ユーザーに動作確認可能な状態として報告して終了
```

## 各ステップの詳細

### 1. 実装

subagent-orchestration の実装エージェント委譲プロンプト雛形をそのまま使う。design.md / impl.md のパスを渡し、範囲外の変更を禁止し、完了後に impl.md を更新させる。

### 2. レビュー

レビューエージェントに以下を必ず渡す:

- 対象タスクの `design.md` / `impl.md` のパス（「まず読め」）
- 今回の実装差分（コミット範囲・変更ファイル一覧・要約）
- 観点: **仕様との乖離**、**バグ・エッジケース**、**可読性・保守性**、**テスト有無**（プロジェクトにテストがある場合）
- 出力形式: 指摘ごとに「該当箇所 / 何が問題か / どう直すべきか（案があれば）/ 分類ヒント（仕様・設計 or 実装）」

レビューエージェントには「指摘 0 件なら 0 件と明記する」ことも要求する。曖昧な「概ね良好」で終わらせない。

### 3. 指摘の分類

各指摘を 2 つに分ける:

| 分類 | 判断者 | 例 |
|------|--------|-----|
| **仕様・設計に関わる** | ユーザー | 「API の返り値の形が design.md と違う」「エラー時の挙動が仕様に書かれていない」「設計方針を変える提案」 |
| **実装判断で閉じる** | Claude | 「変数名を揃えた方がよい」「この if は早期 return にできる」「Result 型でラップし忘れ」「型の書き方が impl.md の既存パターンと不揃い」 |

**判断基準:**

- design.md に書かれた・書かれるべき情報に触れる指摘 → 仕様・設計に分類
- impl.md のパターンや既存コードのローカルな作法・可読性・軽微なバグ・型の細部 → 実装判断
- 迷ったら仕様・設計側（エスカレーション）に倒す。ユーザーの決定なしに仕様を動かさない方が安全

### 4. エスカレーションと記録

**仕様・設計に関わる指摘:**

subagent-orchestration の「エスカレーションプロトコル」に従う。**design.md の「未決事項」に「レビュー指摘 / 選択肢 / トレードオフ / 推奨」を追記**し、口頭では「design.md の該当箇所を見て決定してください」と伝えるだけにする。ユーザーの決定を得たら design.md の「決定事項」に移す。

**実装判断で閉じる指摘:**

**impl.md の「レビュー指摘と対応」セクション**に反復ごとに追記する。フォーマット:

```
### レビュー第 N 周（YYYY-MM-DD）

- **指摘:** <該当箇所と問題の要約>
  - **対応方針:** <採用した対応と理由>
  - **状態:** 対応済み / 対応見送り（理由付き）
```

「対応見送り」を選ぶこともあり得る（コスパが悪い・スコープ外・別タスクにする等）。その場合も理由を書いて残す。

### 5. 対応実装と再レビュー

分類・記録が済んだら、実装エージェントに対応を委譲する。委譲プロンプトには「impl.md の『レビュー指摘と対応』第 N 周の内容を実装せよ」「対応済みに更新せよ」「範囲外の変更はしないこと」を含める。

再度レビューエージェントを回す。前回の指摘が本当に潰れているか＋新規指摘が出ていないかを見る。

### 6. 終了条件

以下がすべて満たされたときのみ、ユーザーに「動作確認可能な状態」として報告して終了する:

- 直近のレビューエージェントからの指摘が 0 件
- 仕様・設計エスカレーションで未解決のものが残っていない（design.md の未決事項が空）
- impl.md のレビュー指摘と対応がすべて「対応済み」または理由付き「対応見送り」で埋まっている

「たぶん大丈夫そう」で止めない。0 件を明示的に確認してから止める。

## 反復の暴走を防ぐ

- **同じ指摘が繰り返し出る:** レビューエージェント側のコンテキスト消費や、対応が不十分な可能性。subagent-orchestration の「サブエージェントのコンテキスト管理」に従い、レビュー・実装いずれも 50% を超える前に新規エージェントに切り替える。
- **枝葉の指摘で無限ループしそう:** レビューエージェントの観点が過剰かもしれない。委譲プロンプトで「今回の差分に閉じた指摘のみ」「スタイルの好み単発は挙げない」等の境界を絞る。
- **指摘対応で範囲外の変更が入る:** 実装エージェントに毎回「範囲外の変更禁止」を明記する。逸脱を見つけたら差し戻し、必要に応じて別タスクとして切り出す。

## 進行報告

反復中も、各周が終わった時点で「第 N 周: 指摘 X 件 / うち仕様関連 Y 件 / 対応済み Z 件」の要約だけをユーザーに出す。指摘の羅列はチャットに貼らず、impl.md / design.md を参照させる。ユーザーに判断してほしいのはエスカレーション事項だけ。

終了報告では「レビュー指摘 0 件を確認済み。impl.md の第 N 周まで対応記録済み。ユーザー動作確認をお願いします」を明示する。
