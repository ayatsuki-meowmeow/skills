# 実装 → レビュー反復ループ — ルール本体

## なぜこのループか

design.md / impl.md が揃っても、一発の実装で仕様に完全準拠し、かつ品質的に問題のないコードが出てくることはまずない。実装エージェントは自分の書いたコードを客観視できないので、独立したレビューエージェントに指摘させ、指摘を潰す反復を回す。

反復を明文化しておかないと、こうなりがち:

- レビュー 1 周で満足して指摘を残したまま「完了」と報告する
- レビュー指摘の中で仕様に関わるものまで Claude が勝手に判断して黙って直す（ユーザーの意思決定を奪う）
- 指摘の履歴が残らず、後から「なぜこの実装になったか」を辿れない
- レビュー収束までは辿り着くが、実装差分をコミットせずに「完了」と報告してしまい、ユーザーがコミット漏れをリカバリするコストを負う

このループの目的は、**レビュー指摘 0 件 + lint 通過 + ユーザーが動作確認できる状態**をゴールに、指摘の分類・エスカレーション・記録・対応を毎周同じ形で回すこと。ゴール到達後はループ内の最終 step として **コミットエージェントに委譲してコミットまで完了させる**。オーケストレーター自身は git 書き込み操作を直接呼ばず、コマンド文字列・分割単位・メッセージ文言はコミットエージェント側に委ねる（詳細は subagent-orchestration の「git 操作の扱い」）。ゴール未達のまま無限ループしないよう **最大 3 周** の上限を設け、超えたら design.md 経由でユーザーに継続判断を仰ぐ。

## 前提

- 該当タスクの `design.md` が存在する（無ければ design-impl-docs スキルに従って先に作る）
- `impl.md` は upfront には要求しない。ループ内で最初に「レビュー指摘と対応」記録などを書くタイミングで作成する（design-impl-docs スキルの作成タイミング規約に従う）
- サブエージェント（Task / Agent ツール）が利用できる
- 本ループは subagent-orchestration スキルの下位ワークフロー。役割分担・委譲プロンプトの共通要素・エスカレーションプロトコル・モデル選択・コンテキスト管理などは subagent-orchestration の rules.md に従う

## ループの全体像

```
[開始] design.md / impl.md 確認、反復カウンタ N = 1
   ↓
1. 実装エージェントに委譲（今回対応する範囲）
   ↓
2. lint チェック（プロジェクトの lint コマンドをオーケストレーターが実行）
   - Fail → 実装エージェントに lint 修正を委譲して 2 を再実行
   ↓
3. レビューエージェントに委譲（実装差分をレビュー）
   ↓
4. 指摘を分類
   - 仕様・設計に関わる指摘 → ユーザーへエスカレーション（design.md 経由）
   - それ以外 → Claude が対応方針を決定
   ↓
5. design.md / impl.md に「レビュー指摘と対応」を記録
   ↓
6. 停止条件を満たす？（指摘 0 件 + lint 通過 + 未決事項なし）
   - Yes → 7 に進む
   - No  → N < 3 なら N++ して 1 に戻る
           N == 3 なら design.md 未決事項に状況を記録してユーザーへエスカレーション
   ↓
7. コミットエージェントに委譲（オーケストレーターは git を直接呼ばない）
   ↓
8. コミット完了を確認してユーザーに動作確認可能な状態として報告し終了
```

## 各ステップの詳細

### 1. 実装

subagent-orchestration の実装エージェント委譲プロンプト雛形をそのまま使う。design.md / impl.md のパスを渡し、範囲外の変更を禁止し、完了後に impl.md を更新させる。

### 2. lint チェック

lint は機械的に検証可能なゴール条件で、レビューエージェントの判断を介さず true/false が決まる。**実行主体は「hook 優先、なければオーケストレーター」**。

**優先: Claude Code hook で走らせる**

プロジェクトの `.claude/settings.json` に PostToolUse / Stop hook として lint コマンドを設定しておくのが第一選択。ハーネス側で機械的に enforce されるため、Claude 側の判断ミスや実行忘れが構造的に消える（harness engineering の考え方に沿う）。hook が失敗すればハーネスが結果を Claude に返すので、そこから修正指示を実装エージェントに委譲する。プロジェクトに lint hook がまだ無い場合は、update-config スキルで settings.json に追加することを検討する。

**fallback: オーケストレーターが直接実行**

hook が整備されていない場合は、実装完了後にオーケストレーターが `pnpm lint` / `npm run lint` / `bun lint` / `biome check` などをプロジェクトの CLAUDE.md か package.json から特定して直接実行する。

**Pass/Fail の扱い（どちらの経路でも共通）:**

- **Pass:** 3. レビューへ進む
- **Fail:** 実装エージェントに「lint エラーの修正のみ」を委譲する（レビュー観点を含めない）。修正後に再度 lint を回し、通ってから 3. に進む
- **Fail が続き修正できない場合:** 「lint 修正 3 回試行しても通らなかった」を design.md 未決事項に記録してユーザーへエスカレーション

lint コマンドが未整備のプロジェクトでは、このステップをスキップしてよい（impl.md にその旨を残す）。ただし typecheck が動くならそれを lint 相当のゲートとして使う。

### 3. レビュー

レビュー実行は **code-review-agent skill** に従う。本ステップは同 skill を invoke するのが原則で、単一エージェントに丸投げしない。code-review-agent は「5 レンズ並列レビュー → Haiku confidence scoring → 閾値フィルタ → 分類ヒント付き指摘の返却」の構造を定義する。

本ステップでオーケストレーターがやること:

- code-review-agent skill に以下を渡す: 対象タスクの `design.md` / `impl.md` のパス、今回の実装差分（コミット範囲・変更ファイル一覧・要約）
- code-review-agent の返却を受け取る: 未解決の疑問（先頭）、`category_hint` 付き指摘のリスト、skip したレンズ（あれば）、各レンズのコンテキスト使用率
- 単発の 1〜数行修正など 5 レンズが過剰なケースでは、code-review-agent を skip して簡易レビュー（1 レンズ相当）で済ませてよい。その判断は明示的に行い、impl.md にも記録する

code-review-agent の詳細は当該 skill の `references/rules.md` 参照。

### 4. 指摘の分類

各指摘を 2 つに分ける:

| 分類 | 判断者 | 例 |
|------|--------|-----|
| **仕様・設計に関わる** | ユーザー | 「API の返り値の形が design.md と違う」「エラー時の挙動が仕様に書かれていない」「設計方針を変える提案」 |
| **実装判断で閉じる** | Claude | 「変数名を揃えた方がよい」「この if は早期 return にできる」「Result 型でラップし忘れ」「型の書き方が impl.md の既存パターンと不揃い」 |

**判断基準:**

- design.md に書かれた・書かれるべき情報に触れる指摘 → 仕様・設計に分類
- impl.md のパターンや既存コードのローカルな作法・可読性・軽微なバグ・型の細部 → 実装判断
- 迷ったら仕様・設計側（エスカレーション）に倒す。ユーザーの決定なしに仕様を動かさない方が安全

### 5. エスカレーションと記録

**仕様・設計に関わる指摘:**

subagent-orchestration の「エスカレーションプロトコル」に従う。**design.md の「未決事項」に「レビュー指摘 / 選択肢 / トレードオフ / 推奨」を追記**し、口頭では「design.md の該当箇所を見て決定してください」と伝えるだけにする。ユーザーの決定を得たら design.md の「決定事項」に移す。

**実装判断で閉じる指摘:**

**impl.md の「レビュー指摘と対応」セクション**に反復ごとに追記する。フォーマット:

```
### レビュー第 N 周（YYYY-MM-DD）

- **指摘:** <該当箇所と問題の要約>
  - **対応方針:** <採用した対応と理由>
  - **状態:** 対応済み / 対応見送り（理由付き）
```

「対応見送り」を選ぶこともあり得る（コスパが悪い・スコープ外・別タスクにする等）。その場合も理由を書いて残す。

### 6. 対応実装と再レビュー

分類・記録が済んだら、実装エージェントに対応を委譲する。委譲プロンプトには「impl.md の『レビュー指摘と対応』第 N 周の内容を実装せよ」「対応済みに更新せよ」「範囲外の変更はしないこと」を含める。

対応実装完了後は必ず 2. lint チェック → 3. レビュー の順で回す。前回の指摘が本当に潰れているか＋新規指摘が出ていないかを見る。この時点で反復カウンタ N を 1 進める。

### 7. コミット（コミットエージェントに委譲）

以下の停止条件をすべて満たしたら、ループの最終 step として **コミットエージェントに委譲** する。

- 直近のレビューエージェントからの指摘が 0 件
- lint が pass している（hook / オーケストレーターいずれの経路でも）
- 仕様・設計エスカレーションで未解決のものが残っていない（design.md の未決事項が空）
- impl.md のレビュー指摘と対応がすべて「対応済み」または理由付き「対応見送り」で埋まっている

「たぶん大丈夫そう」で止めない。0 件を明示的に確認してから委譲に進む。

**オーケストレーターは `git add` / `git commit` を直接叩かない**（subagent-orchestration の「git 操作の扱い」を参照）。コミットエージェントの委譲プロンプトには以下を含める:

- **役割:** 未コミット変更を staging → commit するコミットエージェント
- **コンテキスト:** design.md / impl.md のパス、今回対応した範囲、`git status` / `git diff` の要約（オーケストレーター側で読み取り系を実行して渡す）
- **タスク:** 意味単位で分割してコミットする。design.md / impl.md への追記（「レビュー指摘と対応」記録含む）もコミット対象に入れる
- **規約:** プロジェクトにコミット規約スキル（例: commit-workflow）があればそれに従う。無ければ標準手順（意味単位分割・必要十分なメッセージ・`.env` などアクセス禁止ファイルが staged に混入していないか確認）
- **エスカレーション:** 分割単位・メッセージ文言・staging に含めるか迷ったら自己判断せずユーザーへエスカレーション
- **境界:** コミット以外の変更（コード修正・整形・ファイル追加）は一切加えない

コミット失敗時の扱い:

- **`--no-verify` 等での回避は禁止**（hook をバイパスするのは仕様に反する）
- **hook / pre-commit で lint・format エラー等が出た場合**: 内容が「仕様・設計に関わらない機械的な修正で通せる」なら、実装エージェントに修正のみを委譲して再コミット委譲を試みる。auto-fix（`pnpm lint --fix` / `biome check --write` 等）で閉じる範囲、typo・整形・単純な rule 違反の修正はここに該当する。修正後は必ずステップ 2 の lint チェックを再実行してからコミットに進む
- **仕様・設計に触れる修正が必要な場合**（型定義の意味を変える / API シグネチャを変える / エラーハンドリングの方針を変える 等）: 自力で直さず、subagent-orchestration のエスカレーションプロトコルに従い design.md 未決事項へ載せてユーザーへ返す
- **コミットエージェントが分割・メッセージ等でエスカレーションを返した場合**: 手を止めてユーザーへ伝える（オーケストレーター側で勝手に決めない）
- **同じコミットで hook が 3 回連続で失敗**: それ以上自動修正を続けず、状況を design.md 未決事項に記録してユーザーへエスカレーション（背後に依存不整合や rule 改廃がある可能性）

### 8. 終了報告

コミットエージェントの成果（作成コミットのハッシュ・分割内訳・残った未コミット差分の有無）を確認した上で、ユーザーに「動作確認可能な状態」として報告して終了する。「レビュー指摘 0 件を確認済み。impl.md の第 N 周まで対応記録済み。コミット完了（ハッシュ or 概要）。ユーザー動作確認をお願いします」を明示する。

### 反復上限（3 周）とエスカレーション

反復カウンタ N は「実装 → lint → レビュー → 分類 → 記録」を 1 周として数える。**N == 3 の周で終了条件を満たさない場合は、それ以上 Claude 判断で対応を続けず、design.md 未決事項にユーザー判断を仰ぐ**。

design.md 未決事項に追記する内容:

- **状況:** 3 周回したが「指摘 X 件残 / lint 状態 / 未対応の主要指摘の要約」
- **選択肢:** ①今の指摘方針で更に反復を延長する / ②スコープを縮小して現状を受け入れる / ③設計を見直す（design.md の該当仕様を修正） / ④別タスクに切り出す
- **推奨:** 現状分析に基づく Claude の推奨

その上でユーザーに「design.md 未決事項の該当セクションを見て継続方針を決定してください」と伝えて手を止める。ユーザーの決定が出るまで新たな実装は始めない。同一指摘の周回や lint 修正の詰まりなど、原因が特定できていれば併記する。

## 反復の暴走を防ぐ

**上限は 3 周（詳細は「反復上限（3 周）とエスカレーション」参照）**。これは硬い上限で、超えたら Claude 判断での継続はせず必ず design.md 経由でユーザーに判断を仰ぐ。以下は 3 周に到達する前に打てる手:

- **同じ指摘が繰り返し出る:** レビューエージェント側のコンテキスト消費や、対応が不十分な可能性。subagent-orchestration の「サブエージェントのコンテキスト管理」に従い、レビュー・実装いずれも 50% を超える前に新規エージェントに切り替える。同一指摘が 2 周続いた時点で、原因分析を design.md 未決事項に記録するかどうかを検討する。
- **枝葉の指摘で無限ループしそう:** レビューエージェントの観点が過剰かもしれない。委譲プロンプトで「今回の差分に閉じた指摘のみ」「スタイルの好み単発は挙げない」等の境界を絞る。
- **指摘対応で範囲外の変更が入る:** 実装エージェントに毎回「範囲外の変更禁止」を明記する。逸脱を見つけたら差し戻し、必要に応じて別タスクとして切り出す。
- **lint 修正で詰まる:** 「lint 修正のみ」の委譲を 3 回試して通らなければ、それ自体を design.md 未決事項に上げる（依存不整合・ルール改廃・仕様変更が背後にあることが多い）。

## 進行報告

反復中も、各周が終わった時点で「第 N 周: 指摘 X 件 / うち仕様関連 Y 件 / 対応済み Z 件」の要約だけをユーザーに出す。指摘の羅列はチャットに貼らず、impl.md / design.md を参照させる。ユーザーに判断してほしいのはエスカレーション事項だけ。

終了報告では「レビュー指摘 0 件を確認済み。impl.md の第 N 周まで対応記録済み。コミット完了（ハッシュ or 概要）。ユーザー動作確認をお願いします」を明示する。オーケストレーターは終了 step 直前で `git status` / `git log -1` などを直接読み取り、コミットエージェントの成果報告と齟齬がないことを確認してから報告する。
