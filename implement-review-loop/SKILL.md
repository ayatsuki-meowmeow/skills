# 実装 → レビュー反復ループ

## 目的

design.md 作成後の実装フェーズを、「実装 → レビュー（docs → code sequential） → 指摘分類 → 記録 → 対応」の反復として回し、両レビュー指摘 0 件・lint 通過・未決事項なしまで到達させる（impl.md は初回記録時に生成）。docs レビューを先行させて impl.md drift を即修正することで、code review と次イテレーションの fix 実装が信頼できる impl.md をリファレンスに走れる状態を作る。到達後はループ内の最終 step としてコミットエージェントに委譲してコミットまで完了させ、完了報告する。動作確認依頼は UI・外部影響・仕様確認要のケースに限る。

## 参照

- ルール本体は `references/rules.md` を参照する。実装フェーズに入る前に必ず読むこと。
- 役割分担・委譲プロンプト共通要素・エスカレーションプロトコル・モデル選択・コンテキスト管理は subagent-orchestration スキルに従う（git 書き込み操作の扱い含む）。ドキュメントレビューエージェントの役割定義も同スキル参照。
- design.md / impl.md の編集権限・記載範囲は design-impl-docs スキルに従う。
- ドキュメントレビュー実行（本ループの step 3a）は subagent-orchestration の「ドキュメントレビューエージェント」役に委譲する。impl.md を書いた実装エージェントとは別エージェントで走らせる。指摘が impl.md 修正で閉じるものは 3a 内で即修正、仕様乖離・未決先取りは design.md 経由でエスカレーション（halt）。
- コードレビュー実行（本ループの step 3b、3a-fix 完了後）は code-review-agent スキルに従う（5 レンズ並列 + confidence scoring）。
- レビュー収束後のコミット（本ループの最終 step）はコミットエージェントに委譲する。オーケストレーター自身は git 書き込み系を直接呼ばない（subagent-orchestration の「git 操作の扱い」参照）。プロジェクトにコミット規約スキル（例: commit-workflow）がある場合、コミットエージェントの委譲プロンプトで従わせる。

## 実行手順

1. `references/rules.md` を読む
2. 該当タスクの design.md（と、既にあれば impl.md）を確認し、実装対象範囲を確定する。反復カウンタ N = 1 で開始
3. 実装エージェントに委譲 → lint チェック（hook 優先、なければオーケストレーター実行、fail 時は修正委譲） → **3a. ドキュメントレビュー**（別エージェント、impl.md 未生成 or 構造セクション未変更なら skip）→ 3a 指摘の即対応（impl.md 修正 or エスカレーション halt） → **3b. コードレビュー**（reconciled 済み impl.md をリファレンス）→ 3b 指摘を「仕様・設計」と「実装判断」に分類する
4. 仕様・設計は design.md 経由でユーザーへエスカレーション、それ以外は impl.md の「レビュー指摘と対応」に周ごとに記録して対応を決める（impl.md が未生成なら初回記録時にここで作成）
5. コード修正対応を実装エージェントに委譲して N++ で 3 に戻る。「docs + code レビューとも指摘 0 件 + lint 通過 + 未決事項なし」を確認したらコミットエージェントに委譲してコミット完了を確認し、完了報告して終了（UI・外部影響・仕様確認要のケースのみ動作確認依頼を添える）。N == 3 で終了条件未達なら design.md 未決事項に状況・選択肢・推奨を記録してユーザーへエスカレーション
