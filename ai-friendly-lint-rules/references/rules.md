# AI-friendly Lint Rules — 設計規約

## 目的

カスタム lint ルールを AI 向けに設計するときの規約。「AI がエラーメッセージだけを読んで自力で修正できる」状態をゴールとする。

**なぜこれが要るか**（ループ回数削減の観点）:

- skill / プロンプトだけで規約を伝えると「読み忘れ・忘却・トークン消費・確率的順守」で頭打ちになる
- 静的解析で機械判定可能なルールは lint に降ろせば、違反 1 件あたりのターン数を「レビュー依存の 3〜5 ターン」から「hook 経由の 2 ターン」に圧縮できる
- ただしメッセージが AI 向けに設計されていないと、hook で返しても AI が意図を汲み損ねて 3〜5 ターンに戻る

このスキルは **ルール中身の設計原則** を扱う。実行環境や配置手順（lint ラッパー、モノレポでの扱い、pre-commit 連携など）は各プロジェクトのローカルスキルで扱う。

## 対象ツール

ツール非依存の設計原則。以下はいずれも同じ設計を当てはめられる:

- ESLint (JS/TS, `meta.messages` + `context.report`)
- oxlint / Biome (Rust 製 JS/TS linter, diagnostic API)
- Ruff (Python, ruff rule 実装)
- Clippy (Rust, `declare_lint_pass!`)
- RuboCop (Ruby, `Cop` DSL)
- PMD / SpotBugs (Java)
- 自作 AST スクリプト（tree-sitter / ts-morph / rustc plugin / SwiftSyntax 等）

ツール固有の書式（`messages` の書き方、autofix の API 名）は違うが、**AI に向けたメッセージ設計の 4 要素と分類判断は共通** で、そこがこのスキルの対象。

## 1. lint 化するか判断する — ルール分類 A〜E

コーディング規約を機械判定可能性で 5 分類する。A〜D は静的解析化を検討する価値があり、E は skill に残す。

### 分類 A: 禁止パターン（AST マッチで完全に検出可）

「特定の構文・API を使わせない」タイプ。AST の形だけで判定できる。

**典型例:**
- 型システムの脱出口（型キャスト構文・任意型・型エラー抑制コメント）の禁止
- 特定 API・ライブラリ関数の呼び出し禁止
- 直接比較演算子（`=== null` 等）の禁止と専用ヘルパー関数への強制

**lint 化コスト**: 低（AST マッチのみ）。実装は数十行の visitor で書ける。

### 分類 B: 必須パターン（AST + 型情報で検出可）

「ある文脈では必ずこう書け」タイプ。AST に加え、型情報が必要な場合が多い。

**典型例:**
- 公開関数の引数・戻り値の型注釈を必須にする
- 配列型のイミュータブル既定化（`readonly T[]` 等）
- `undefined` を返しうる式に coalesce を補完（`?? null` 補完）

**lint 化コスト**: 中。型情報付き lint（TS なら `projectService`、Rust なら type-check pass）が要るので実行時間はやや長い。ただし hook 経由 per-file 実行なら実用範囲。

**autofix と相性が良い**: 書き換えパターンが決定的なので autofix にしやすい。AI 側の修正ターンすら不要になる。

### 分類 C: 命名規則（正規表現または型情報で検出可）

**典型例:**
- 例外変数・DOM 要素・イベントオブジェクト・for-of 変数などの短縮名統一
- モック変数の命名慣習（`vi.fn()` 代入先の末尾 `Mock` など）
- ファイル名の kebab-case 統一
- 特殊用途ファイルの命名慣習（`*.mock.ts` / `*.stories.tsx` / `*.snap` 等）
- 環境変数のプレフィックス

**lint 化コスト**: 低〜中。大部分は正規表現ベース。型情報が要るのは `vi.fn()` 代入検出のようなパターンのみ。

**副次効果**: レビュー時に「命名の議論」が発生しなくなる。人間レビュアーも AI レビュアーも命名指摘ターンを消費しなくなる。

### 分類 D: 構造・配置（ファイルパス・import グラフで検出可）

**典型例:**
- 1 ファイル 1 エクスポート関数の徹底
- import 経路の制約（path alias 禁止・相対パス統一）
- 型のみの import を `import type` に統一
- テストファイルと実装ファイルの 1:1 配置
- ディレクトリ境界を跨ぐ import 禁止（レイヤードアーキテクチャの強制）

**lint 化コスト**: 中。ファイルパス判定と import グラフ解析が要る。ESLint なら `no-restricted-imports` や独自プラグイン、Rust なら `clippy::module_name_repetitions` 系。

**効果**: 「どこに置くか」で AI が迷うシチュエーションを減らせる。アーキテクチャ判断の一部を機械化できる。

### 分類 E: 意味判定必須 — skill 領域（lint 化しない）

以下は静的解析化を試みるべきでない。skill に残す:

- **設計判断**: どこに責務を分けるか、どの層に共通化するか、Props の設計方針
- **ワークフロー規約**: git ブランチ運用、コミット粒度、PR 分割
- **粒度判断**: ファイル分割の粒度、抽象化のタイミング、テストの観点設計
- **ユーザー確認を要するもの**: 仕様変更・破壊的変更・危険な操作
- **文脈依存の禁止事項**: 「基本 X だが特定領域では Y」のような場合分けが強い規約

「skill で書いても違反率が下がりにくい」もしくは「静的解析では表現できない」の少なくとも一方に該当するものが分類 E。

### 判定チェックリスト

新規ルール案を出す前に、以下 3 問全部 YES なら A〜D 候補、1 つでも NO なら E に寄せる:

1. 違反 / 非違反が **AST / 型 / ファイルパス** だけで判定できるか
2. 誤検出（false positive）が出た場合、**逃げ道**（disable コメント等）で無理なく回避できるか
3. **修正の書き換え** をエラーメッセージだけで AI に伝えられるか（決定的な代替案があるか）

## 2. エラーメッセージ設計 — 本スキルの主眼

lint エラーメッセージは、単なる「違反です」ではなく **AI が次に取るべき行動を含めて書く**。AI がメッセージを読んで即修正できれば、追加ターンなしで解決する。

### 4 要素

すべてのメッセージは次を含める:

1. **何がダメか** — 検出内容を 1 文で
2. **なぜダメか** — 1 行の理由（省略可: 自明なとき）
3. **どう直すか** — 具体的な書き換え例 or 代替 API
4. **参考規約名** — プロジェクト規約ドキュメント名 / タグ

### 良い例 vs 悪い例

**悪い例（ツール既定風）:**
```
Unsafe expression assigned to variable.
```

**良い例（AI 向け）:**
```
undefined を返しうる式は `?? null` で null に統一してください (プロジェクト規約 / null-undefined-normalization).
(autofix で自動修正されます)
```

**さらに良い例（4 要素を明示）:**
```
`undefined` を返しうる式が変数に格納されています。
プロジェクト規約により null と undefined を混在させないため、`?? null` で null に寄せてください。
例: `const value = maybe()` → `const value = maybe() ?? null`
規約: プロジェクト規約 / null-undefined-normalization
```

### 書き方の指針

- **skill の "代替手段の表" をメッセージにコピーする感覚で書く**。skill を読まなくても、AI は hook で受け取ったメッセージ 1 発で修正できるべき
- **日本語で書く**（プロジェクトの主要言語がそうなら）。AI は日本語も英語も同等に読めるが、レビュアー・利用者向けに合わせる
- **メッセージ末尾に規約名を明記する**。例: `(プロジェクト規約 / team-coding)` / `(PEP 8 §naming)` / `(rustc allowlist)`
- **autofix がある場合は「autofix で自動修正されます」と書く**。AI は待たずに次に進める
- **主観語を避ける**（"よくない" / "推奨されない"）→ **決定的な語** を使う（"禁止" / "統一" / "必須"）

### 既存ルールへのマッピング例

`no-undefined-in-variable` 相当ルールの messages を 4 要素で分解:

| 要素 | 該当箇所 |
|---|---|
| 何がダメか | `undefined を返しうる式は` |
| なぜダメか | (省略: 自明として省いている) |
| どう直すか | `` `?? null` で null に統一してください `` |
| 参考規約名 | `(プロジェクト規約 / team-coding)` |

改善余地: 「なぜ null に統一するか」を 1 行足す、autofix 有無を明記する、あたり。

### AST 位置情報の付け方

多くの lint ツールは `context.report({ node, ... })` のように **報告対象ノード** を指定できる。**メッセージ本文で "この式" と書くだけでなく、必ず該当ノードを指定** して、エディタ／ hook 側でハイライト位置が取れるようにする。

- ESLint: `context.report({ node: init, messageId: '...' })`
- Clippy: `span_lint(cx, LINT, expr.span, "message")`
- Ruff: `Diagnostic::new(...).with_range(range)`

## 3. 例外パスの設計

すべてのルールには「破りたい場合」がある。lint 化する際は、この例外パスも設計する。3 段階から選ぶ:

### 3-1. 絶対禁止

`disable` コメントも許さないもの。ルールに例外を書かない。

**適用条件:**
- 違反が **常に** 事故であり、正当な理由が想定できない
- 例: 型システムの脱出口の完全禁止、`--no-verify` の禁止

**運用:**
- disable コメントを検出したら別のルールで弾く（ESLint なら `eslint-comments/no-use`）

### 3-2. 条件付き許容

特定パスや特定パターンでのみ許容。lint 設定側の `overrides` で除外する。

**適用条件:**
- 違反が **特定ディレクトリでは正当**（例: scratch / migration / generated code）
- 違反が **特定 API では避けられない**（例: 外部ライブラリの型定義に合わせる）

**運用:**
- ルール本体には例外を書かない。**設定側で override** する
- ルールの docs に「override 想定パス」を明記する

### 3-3. 明示的な逸脱

ケースバイケースで許容。`disable` コメントに **理由を必須** にする。

**適用条件:**
- 違反が **稀に正当** だが、正当性はレビュアーが個別判断する必要がある
- 例: readonly 徹底ルールで、パフォーマンス上どうしても mutable が必要な特定関数

**運用:**
- `disable` コメントに理由を必須にする（ESLint なら `eslint-comments/require-description`、Ruff なら `noqa` に理由テキスト）
- 理由テキストの規約: `<disable-directive> <rule-name> -- <理由 1 行>`

### どの段階を選ぶかの判断

- **迷ったら 3-3（明示的な逸脱）** を既定にする。運用しながら「常に事故」なら 3-1 に上げ、「特定パスで常に許容」なら 3-2 に下げる
- **新規ルールを 3-1（絶対禁止）で入れるのは慎重に**。想定外の正当ケースで AI・人間の両方が詰まる

## 4. autofix を付けるかの判断

autofix があると AI の修正ターンすら省ける。ただし付けるべきでないケースもある。

### 付けてよい条件（すべて満たすとき）

1. **書き換えが決定的** — 入力 → 出力が 1 対 1 で決まる
2. **意味論を変えない** — 実行時挙動が同じ、または「変わるが元コードが誤りなので変わってよい」
3. **型として valid** — autofix 後にコンパイルが通る
4. **可読性を大きく損なわない** — 括弧の増減など、些細な範囲

### 付けるべきでないケース

- 書き換え候補が **複数** ある（"どちらを選ぶ" を AI に任せたい）
- 書き換えが **文脈判断** を要する（周辺コードの意図を汲む必要がある）
- **命名変更** など、呼び出し側の追随が要る（fixer では変えられない範囲がある）

例:
- `no-undefined-in-variable` は autofix あり（`(expr) ?? null` が決定的）
- `mock-variable-naming` は autofix なし（呼び出し側の追随が要る）

### 検証手順

autofix を付ける場合は必ず:

1. NG ケースの scratch サンプルを用意
2. autofix 前後で diff を取る
3. **型として valid** で **意味論を変えない** ことを目視確認

## 5. 移行判断のトレードオフ

skill → lint 移行を無条件で進めるわけではない。以下のトレードオフを見た上で判断する:

| 観点 | skill 有利 | lint 有利 |
|---|---|---|
| 記述コスト | プロンプトは自然言語で書ける | AST を扱う必要があり初期実装が重い |
| 変更容易性 | プロンプトを 1 行足せば済む | ルール更新にコード変更 + テストが要る |
| 実行速度 | 追加時間ゼロ | 型情報付き lint は数秒〜数十秒 |
| 誤検出 | 曖昧な指示は AI が状況判断できる | false positive は AI をハマらせる |
| 網羅性 | AI 依存で穴が空く | 決定的に網羅 |
| 拡張性 | ルール数増加でプロンプト圧迫 | ルール数増加は無影響（実行時間は増える） |
| チーム共有 | AI 用途に閉じる | チーム全体・IDE・CI で共有可能 |

**「網羅性 × 拡張性 × チーム共有」で lint が有利**、**「初期記述コスト × 変更容易性 × 誤検出耐性」で skill が有利**。

### 移行の優先度

| 優先度 | 特徴 | 例 |
|---|---|---|
| 高 | 違反頻度が高く、機械判定可能で、autofix 可能 | null/undefined フォールバック / 型注釈省略 |
| 中 | 違反頻度が高く、機械判定可能だが autofix 不可 | 命名規則 / モックファイル名 |
| 低 | 違反頻度は低いが、機械判定可能 | 環境変数プレフィックス / 特殊 import 順序 |
| skill に残す | 意味判定が必要 / 文脈依存 / ワークフロー系 | ファイル分割粒度 / 設計方針 / コミット粒度 |

**フィードバックループ**: 「AI がミスる → レビューで指摘 → ミスの型を分析 → 分類 A〜D に該当すれば lint 化」を回すことで、skill は徐々に薄くなり、lint ルールは徐々に厚くなる。

## 6. 設計チェックリスト（新規ルール実装前）

新規ルールを 1 つ書く前に、以下 8 項目を全部埋める。埋まらないものがあれば、そのルールは lint 化に向いていないか、設計が未熟。

1. [ ] **分類**: A / B / C / D のどれか（E ならこのスキルの対象外）
2. [ ] **検出対象**: どの AST ノード / パス / 型を見るか
3. [ ] **NG ケース**: 具体コード 1〜3 個
4. [ ] **OK ケース**: 除外条件を示す具体コード 1〜3 個（特に「よく間違えて flag しがちな正当ケース」）
5. [ ] **エラーメッセージ**: 4 要素すべて含む
6. [ ] **例外パス**: 絶対禁止 / 条件付き許容 / 明示的な逸脱 のどれか
7. [ ] **autofix**: あり / なし + 理由
8. [ ] **参考規約**: 元になる skill / ドキュメントへのリンク

このチェックリストを埋めた状態で実装に入る。ツール固有の手順（配置場所・plugin 登録・config 有効化）は各プロジェクトのローカルスキルに委譲する。
